● Honest take: forking is the fastest path to a working codex-toolbox, but it's
  the wrong move if you plan to maintain both.

  The real cost of a fork is skill drift. Skills are the core value of this repo
   — design, implement, review-code with its profiles, chain-of-verification,
  etc. They evolve: you tweak detection logic, improve prompts, add profiles,
  fix workflow ordering bugs. In a fork, every improvement has to be manually
  ported. That discipline lasts about two weeks before one repo falls behind.

  What a fork actually looks like:
  - Copy repo, gut .claude/, .claude-plugin/, replace with .codex/,
  .codex-plugin/
  - Rewrite every SKILL.md to use codex tool names instead of Claude's
  - Replace hooks.json (Claude format) with hooks.json (codex format — different
   schema, different blocking mechanism)
  - Replace CLAUDE.md with AGENTS.md
  - Delete commands (codex has no equivalent — skills serve that role)
  - Rewrite agents as codex subagent definitions
  - Rewrite all tests

  That's not "refactor," that's a rewrite. And then you maintain two repos where
   70%+ of the content (skill logic, profiles, shared instructions) is
  identical.

  The lighter alternative that avoids both forking AND the full opencode-style 
  restructure:

  1. Move klaude-plugin/skills/ to skills/ at repo root (one git mv, update
  symlink + marketplace path)
  2. Keep klaude-plugin/ for all other Claude artifacts (commands, agents,
  hooks, scripts) — don't rename it
  3. Add plugins/codex/ with .codex-plugin/plugin.json, codex hooks.json,
  AGENTS.md template
  4. Codex plugin's skills/ symlinks to ../../skills/ (or copies at install time
   if codex doesn't follow symlinks)
  5. Skip the provider-neutral SKILL.md rewrite entirely — codex's model is
  smart enough to map tool names if the SessionStart hook injects a mapping
  table

  Step 5 is the key insight: instead of rewriting every SKILL.md to say "read
  the file" instead of Read, just inject a tool-name mapping via the
  SessionStart hook's additionalContext. Codex's model sees "When this skill
  says Read, use read_file; when it says Bash, use shell" and it works. Impure,
  but it avoids touching every skill file.

  Total repo changes: one directory move, one new plugins/codex/ tree, zero
  skill rewrites. Skills stay single-source. What do you think?

● Section 5: Capy Enforcement via PreToolUse Hooks

  Claude enforces capy routing in two independent mechanisms: validate-bash.sh
  (file-path denylist) and .claude/capy/CLAUDE.md (HTTP/output routing injected
  as context). Codex reproduces both via PreToolUse hooks in .codex/hooks.json,
  with a documented enforcement gap.

  What we CAN enforce (PreToolUse on Bash):

  The hook script receives the shell command on stdin as tool_input.command. It
  checks two categories and blocks by emitting permissionDecision: "deny" JSON
  on stdout:

  1. File-path denylist (port of validate-bash.sh): blocks access to .env,
  .ansible/, .terraform/, build/, dist/, node_modules, __pycache__, .git/,
  venv/, .pyc, .csv, .log. Same FORBIDDEN_PATTERNS array, ported to the hook
  script.
  2. Capy HTTP routing: blocks curl/wget commands and inline-HTTP patterns
  (fetch('http, requests.get(, requests.post(, http.get(, http.request(). Deny
  reason directs to capy equivalents.

  What we CANNOT enforce (documented gap — ADR required):

  Codex's PreToolUse currently only intercepts Bash/shell. It cannot intercept
  read_file, write_file, apply_patch, web_search, or MCP tool calls. This means:

  - File-path denylist on read_file is advisory only (via SessionStart context)
  - WebFetch-equivalent blocking is advisory only (codex has no web_fetch tool,
  but web_search can't be hooked)
  - MCP tool interception is not possible

  This is codex's current limitation (PreToolUse docs explicitly say "Work in
  progress"). The ADR documents: the gap exists, it's acceptable because it's
  the only option, and when codex expands hook coverage we can add enforcement.
  The SessionStart context provides the advisory layer for everything hooks
  can't enforce.

● Section 7: Starlark Rules (Command Policies)

  Claude uses flat allow/deny lists in .claude/settings.json for permission
  control. Codex uses Starlark .rules files — a more powerful mechanism with
  pattern matching, justifications, and inline test cases.

  The port creates .codex/rules/default.rules translating Claude's deny list
  into prefix_rule() calls:

  # Destructive operations
  prefix_rule(
      pattern = ["rm"],
      decision = "deny",
      justification = "Destructive file deletion blocked by policy",
      match = ["rm -rf /tmp/build"],
      not_match = ["rmdir empty-dir"],
  )

  prefix_rule(
      pattern = ["git", "push", "--force"],
      decision = "deny",
      justification = "Force push blocked by policy",
  )

  What gets ported from Claude's deny list: rm, git push --force, git reset 
  --hard, and any other destructive commands currently denied in
  .claude/settings.json. Each becomes a prefix_rule with decision = "deny".

  What gets ported as "prompt": Commands that Claude marks as "ask" — operations
   that aren't blocked but require user confirmation. These map to decision = 
  "prompt" in Starlark, which surfaces an approval prompt in codex's UI.

  Expansion opportunity: Since Starlark is more expressive than flat lists, we
  can add rules Claude can't express — e.g., denying git push to main/master
  specifically while allowing pushes to other branches. This is a future
  enhancement; the initial port matches Claude's deny list 1:1.

  Testing: Rules can be validated offline with codex execpolicy check --pretty 
  --rules .codex/rules/default.rules -- <command>, which emits JSON showing the
  matched rule and decision. This goes into the verification steps.
